Implemented

Make each eval case trace explain what the agent did

Add a harness-owned span tree beneath MLflow's native case trace while retaining the separate eval-only raw OTEL stream. Normalize CLI tool, model, timing, and token evidence so instruction-following assessments can inspect agent and subagent behavior without navigating hundreds of transport traces.

01 Scope and success

OutcomeAcceptance evidence
Readable causal traceOne predict_fn trace contains workspace preparation, agent invocation, normalized CLI events, evidence capture, and cleanup spans where applicable.
Agent-call evidenceTool and collaboration events expose tool name, status, requested agent/model hints when emitted by the CLI, and safe summary fields.
Operational metricsEvery successfully completed predictor case records wall-clock completion seconds and available input, cached input, reasoning, output, and total tokens as diagnostic MLflow feedback.
Raw telemetry preservedAlloy continues accepting only telemetry.purpose=evaluation traces on the dedicated receiver, redacting known identity fields, and forwarding the low-level stream.
Bounded validation costStatic checks plus one selected single-arm agent eval; no treatment/control campaign and no judge unless the chosen case requires one.

Codex / Claude raw OTEL ── evaluation tag ── Alloy ── MLflow diagnostics
The logical execution ID remains the join key between the readable harness trace and raw runtime traces. Alloy is a routing and privacy boundary, not the place to reconstruct application semantics.

03 Implementation sequence

  1. Extend the CLI result contract with normalized events, token usage, usage source, and measured subprocess duration.
  2. Parse Codex JSONL and Claude stream JSON conservatively. Preserve unknown or unavailable fields explicitly; deterministically derive documented totals but never estimate missing counts.
  3. Create explicit MLflow child spans around harness phases and render normalized events as children of agent.invoke.
  4. Return operational evidence with predictor output and emit universal diagnostic feedback for completion time and token dimensions.
  5. Document the two trace layers and update the evaluation-library codemap.
Privacy and payload constraints Store normalized names, status, durations, numeric usage, and summaries bounded to 1,000 characters. Do not duplicate complete prompts, responses, patches, command output, authentication material, or arbitrary raw CLI event payloads into child span attributes. Full normalized shell commands remain in the predictor output used by deterministic scorers.

04 Verification

Run Python compile, formatter/linter/type checks, Compose configuration validation, and the HomeOps build without invoking unit tests. Start the owned MLflow and Alloy services, run one cheap single-arm case, query the resulting trace through MLflow, and confirm nested span names plus duration/token feedback. Stop only services started for this verification.

05 Risks and decisions

RiskDecision
CLI schemas differ or evolveUse profile-specific adapters into a small shared contract and retain an explicit usage source/version.
Post-hoc spans misrepresent durationUse MLflow's explicit nanosecond start/end APIs when source timing exists; otherwise label events as observations and store reported duration as an attribute.
Token totals are not comparableStore dimensions separately, define total deterministically only from emitted fields, and disclose unavailable categories.
Raw OTEL volume remains noisyLeave routing unchanged in this slice; normalized traces become the primary inspection surface while raw traces remain secondary diagnostics.
